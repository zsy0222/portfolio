## 动态后向切片及其局限

动态后向切片（Dynamic Backward Slicing）是调试中最常用的切片方法：给定错误输出点，沿数据依赖与控制依赖向后追溯，得到所有可能影响该错误的语句集合。

核心局限：切片是依赖关系的传递闭包，无法在找到 bug 时自动停止，会一直追溯到输入/初始状态。切片结果能缩小搜索范围（如从 10,000 行缩到 500 行），但不会标注 bug 位置。

## Chop：双向切片约束范围

Chop 通过引入第二个锚点来约束追溯范围：

- **Source**：怀疑的源语句 / 输入 / 赋值
- **Sink**：错误输出 / 故障表现

Chop 不是简单取前向与后向切片的交集，而是 PDG 上的**受限可达性问题**（Jackson & Rollins, 1994）。传统 PDG 的缺陷在于缺乏逻辑连接——节点 A 可能在正向切片中是因为一条路径，在反向切片中是因为另一条完全不同的路径，而这两条路径可能根本没有连接起来。

## 断言作为决策点

定位方法的核心思路：先进行因果链诊断，再二分定位矛盾，结合源头输出核对与 SBFL/LLM 外部辅助。

> 后向只找"语法上可能影响"的语句，前向过滤"语义上永远不可能执行"的路径。

## 方向转变：从传统 PDG 到 LLM 切片引擎

此前思路：开发传统 PDG 切片工具（Chop4LLM），产出切片结果喂给 LLM 分析。

新思路：不写传统 PDG 切片算法，而是设计一套**结构化切片推理工作流范式**（Slicing Workflow Paradigm），让 LLM 自己按范式推理依赖关系、输出切片代码。

核心变化：旧方案是 PDG 算法做切片 + LLM 做分析（两个独立步骤）；新方案是 LLM 自身成为切片引擎，范式是操作手册（一个统一框架）。

## 关键洞察：LLM 语义切片 vs 传统 PDG 可达性

传统方法的核心瓶颈在于类型决议和指针分析是全程序的、O(n³) 的操作，有巨大的入场费，且无法处理不可编译代码。

LLM 的优势：绕过类型决议、不可编译代码也能切、语义消歧、按需付费、语言无关、可解释。

LLM 的劣势：不 sound、不确定、长程序受 context window 限制、成本不可预测。

> 一句话总结：传统方法用数学精确性换取工程不实用；LLM 用语义推断换取工程实用。Debug 场景天然容忍不完美——不需要 soundness，需要一条合理的因果链。
